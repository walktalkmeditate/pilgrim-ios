## Confirmed findings


### CRITICAL

- AF1. **Walk-end flow deletes the crash-recovery checkpoint before the save transaction is confirmed — a failed save loses the walk irrecoverably**
  - `Pilgrim/Scenes/ActiveWalk/ActiveWalkViewModel.swift:231` · lens: data-safety
  - ActiveWalkViewModel.stop() calls sessionGuard?.stopAndCleanup() (which deletes walk_checkpoint.json, WalkSessionGuard.swift:121) before builder.setStatus(.ready) (line 236) triggers snapshot creation → onWalkCompleted → DataManager.saveWalk (MainCoordinatorView.swift:62, async CoreStore transaction). If that save fails, MainCoordinator only sets showSaveError = true (MainCoordinatorView.swift:75-77); pendingSnapshot is never set, the builder has already reset() its relays (WalkBuilder.swift:173), and the checkpoint that contained the full walk was already deleted — the multi-hour walk is gone with no retry path. Its audio files, now referenced by no DB row, are deleted by OrphanRecordingSweep at next launch. The checkpoint system exists exactly for this resilience but is torn down before the commit is confirmed. (cancel() at line 242 doing the same is intentional — user discard.)
  - Fix sketch: Split stopAndCleanup into stop (timer/observers/power) and deleteCheckpointFile; call the checkpoint deletion only inside the saveWalk success callback in MainCoordinator (mirroring WalkSessionGuard.recoverIfNeeded, which deletes the checkpoint only after save succeeds, line 368-377). On failure, keep the checkpoint so launch recovery can re-save the walk.
  - Verifier confidence: 0.88, 0.88


### MAJOR

- AF2. **Launch race: OrphanRecordingSweep can delete a crashed walk's audio files before WalkSessionGuard.recoverIfNeeded commits them to the database**
  - `Pilgrim/AppDelegate.swift:115` · lens: error-reliability
  - AppDelegate fires RecordingPathRecovery.run { OrphanRecordingSweep.run() } immediately after setting appLaunchState = .done, while WalkSessionGuard.recoverIfNeeded runs independently from MainCoordinator.init (MainCoordinatorView.swift:22-26), which only happens when SwiftUI builds MainTabView on a later runloop tick. OrphanRecordingSweep deletes every .m4a/.wav under Recordings/ that is not referenced by a VoiceRecording row (OrphanRecordingSweep.swift:42-48). After a mid-walk crash, the crashed walk's recordings exist only in the checkpoint file — their DB rows are created by recoverIfNeeded's DataManager.saveWalk (WalkSessionGuard.swift:368), an async CoreStore transaction. If RecordingPathRecovery's transaction completes and its main-queue success block runs before SwiftUI instantiates MainCoordinator (plausible with a small DB), the sweep's fetch is enqueued before the recovery save, sees no rows for those files, and permanently deletes the crashed walk's voice recordings — exactly the data the recovery system exists to protect. Nothing serializes recovery before the sweep.
  - Fix sketch: Order the launch chain explicitly: run RecordingPathRecovery + OrphanRecordingSweep only after WalkSessionGuard.recoverIfNeeded's completion (recovery already has a completion callback), or have the sweep skip the walk-UUID directory referenced by an existing checkpoint file.
  - Verifier confidence: 0.85, 0.75

- AF4. **AudioSessionCoordinator never downgrades the session mode while any consumer remains — playAndRecord persists after voice notes end**
  - `Pilgrim/Models/Audio/AudioSessionCoordinator.swift:34` · lens: resource-leaks
  - deactivate() only applies .idle when activeConsumers becomes empty; otherwise currentMode keeps whatever the last activate() set. During a walk with a soundscape playing ("soundscape" consumer held), starting a voice note applies .recordAndPlay (playAndRecord + defaultToSpeaker). When the recording ends, removing "voiceRecording" leaves the session in playAndRecord — a mic-active category — until the next bell/whisper/voice-guide activate() happens to re-apply .playbackOnly, which on a quiet walk can be tens of minutes. For a privacy-first app, an input-capable audio session held open after recording stops is a meaningful resource/perception issue (input route stays powered).

- AF5. **Soundscape never resumes after an audio interruption; SoundscapePlayer.isPlaying stays true while silent for the rest of the walk/meditation**
  - `Pilgrim/Models/Audio/AudioSessionCoordinator.swift:54` · lens: error-reliability
  - When an interruption ends, AudioSessionCoordinator.handleInterruption only re-applies the session category/active state (`applyMode(currentMode)`); it does not notify any consumer. AVAudioPlayer does not auto-resume after an interruption, so a phone call, Siri query, or alarm during a meditation/walk leaves SoundscapePlayer.activePlayer paused forever.   - Fix sketch: Have AudioSessionCoordinator publish interruption began/ended events (or post a custom Notification). - AF6. **VoiceGuidePlayer.stop() drops onFinished without invoking it — scheduler isPlaying latch sticks and the voice guide goes permanently silent**
  - `Pilgrim/Models/Audio/VoiceGuide/VoiceGuidePlayer.swift:56` · lens: resource-leaks
  - VoiceGuideScheduler sets `isPlaying = true` via markPlaybackStarted() before onShouldPlay (VoiceGuideScheduler.swift:138-139), and the ONLY thing that clears it is markPlayed(), which is called from VoiceGuidePlayer's onFinished callback. But VoiceGuidePlayer.stop() nils onFinished without invoking it, and the play() error path (lines 49-53) does the same. Any stop while a prompt is mid-playback permanently wedges the scheduler: tick() forever bails on `!isPlaying`. - AF7. **Walk deletion relies on CoreStore's default .nullify delete rule — tended-import overwrite orphans every child row (route samples, recordings, pauses) in the database forever**
  - `Pilgrim/Models/Data/PilgrimPackage/PilgrimPackageImporter.swift:278` · lens: data-safety
  - No relationship in the schema specifies a deleteRule (PilgrimV7.swift:140-148 and each child's _workout = Relationship.ToOne, e.g. line 240); CoreStore's default is .nullify (Pods/CoreStore/Sources/Relationship.ToOne.swift:68). So deleteExistingWalksMatching's transaction.delete(walk) only nullifies the children's workout pointer: every RouteDataSample (often thousands per walk), VoiceRecording, WalkPause, WalkEvent, ActivityInterval, Waypoint and WalkPhoto row of the replaced walk stays in the SQLite store, unreachable and never cleaned (deleteAll wipes by entity type, but nothing else does). Each tended import re-orphans a full child set per replaced walk — compounding DB bloat. The orphaned VoiceRecording rows also keep their fileRelativePath values, which permanently pin those paths in OrphanRecordingSweep's referenced set (it fetches ALL VoiceRecording rows, OrphanRecordingSweep.swift:25-26), so the audio files become unsweepable even when they should be removable. The same gap exists in the unused public API DataManager.deleteObject (DataManager.swift:866), which deletes a Walk and its audio files but leaves all child rows behind.
  - Fix sketch: In deleteExistingWalksMatching, explicitly delete each child collection before deleting the walk (mirror stripHeavyData, importer lines 425-432), and do the same in DataManager.deleteObject. Alternatively set deleteRule: .cascade on the child-side _workout ToOne relationships — delete rules are not part of CoreData's entity version hash, so this should not require a store migration (verify with a migration test before relying on it).
  - Verifier confidence: 0.85, 0.9

- AF8. **WalkBuilder leaks permanently: preSnapshotFlush closures in LocationManagement and StepCounter capture the builder strongly**
  - `Pilgrim/Models/Walk/WalkBuilder/Components/LocationManagement.swift:154` · lens: resource-leaks
  - LocationManagement.init and StepCounter.init (StepCounter.swift:106-109) register closures into builder.preSnapshotFlushActions that capture the `builder` init parameter strongly ([weak self] covers only the component). preSnapshotFlushActions is never cleared (WalkBuilder.swift:64-69), so WalkBuilder retains a closure that retains WalkBuilder — a self-retain cycle.   - Verifier confidence: 0.92, 0.92

- AF9. **Per-GPS-sample O(n) pipeline: full route array copy, full coordinate remap, full segment rebuild, and full-route GeoJSON re-upload — cost grows linearly with walk length**
  - `Pilgrim/Models/Walk/WalkBuilder/Components/LocationManagement.swift:242` · lens: performance
  - With kCLLocationAccuracyBest and kCLDistanceFilterNone, location callbacks arrive ~1/s while walking. For each accepted sample: (1) LocationManagement copies the entire samples array (locationsRelay.accept(value + [sample])); (2) ActiveWalkViewModel's locations sink (ActiveWalkViewModel.swift:198) maps ALL samples to a new CLLocationCoordinate2D array; (3) buildActivitySegments (ActiveWalkViewModel.swift:202, 504-530) iterates ALL samples, calling activityType(at:) which scans meditationIntervals and completedRecordings for each sample — O(n·m); (4) the published routeSegments cause PilgrimMapView.applyRouteSource to rebuild the full FeatureCollection and call updateGeoJSONSource with the entire route (PilgrimMapView.swift:288-301). All of this runs on the main thread. At hour 3 of a walk (~10,000 samples) every new GPS fix triggers several 10k-element array allocations, a 10k-iteration segment rebuild with nested interval scans, and a full-route GeoJSON serialization/upload into Mapbox — a per-second cost that compounds with walk duration, exactly the pattern CLAUDE.md flags for this app.
  - Fix sketch: Append incrementally: keep routeCoordinates and the last open segment in the view model and only extend them with the new sample(s) (activity type only changes at talk/meditate boundaries). For Mapbox, append to the GeoJSON source (partial update / per-segment sources) instead of re-uploading the whole route, or throttle full uploads to every N samples.
  - Verifier confidence: 0.75, 0.85

- AF11. **Talk recording silently stops capturing audio after a non-connected-call audio interruption (declined call, Siri, alarm) — UI keeps showing recording**
  - `Pilgrim/Models/Walk/WalkBuilder/Components/VoiceRecordingManagement.swift:265` · lens: error-reliability

- AF12. **WalkSessionGuard battery/thermal sinks run off-main; checkpoint Timer gets rescheduled on a runloop-less thread and silently stops firing**
  - `Pilgrim/Models/Walk/WalkSessionGuard.swift:99` · lens: concurrency
  - `ProcessInfo.thermalStateDidChangeNotification` is posted on a background thread (and `UIDevice.batteryLevelDidChangeNotification` is not guaranteed main), but neither NotificationCenter sink in `start()` has `.receive(on: DispatchQueue.main)` — note the adjacent `viewModel?.$isMeditating` sink (line 106) DOES have it. When a tier change arrives on a background thread, `recalculateTier()` → `applyTier()` → `restartTimer()` runs there: `checkpointTimer?.invalidate()` is called from a thread other than the one that installed the timer (documented as illegal for Timer), and `Timer.scheduledTimer` (line 208) attaches the replacement to the current GCD worker thread's run loop, which never runs — so the new timer NEVER fires. Crash-recovery checkpointing silently stops for the remainder of the walk; if the app is later killed (the exact scenario checkpointing exists for), everything after the last checkpoint is lost. Thermal/battery tier transitions are routine on long summer walks (battery dropping through 20%, thermal fair→serious). `recalculateTier` also reads `viewModel?.isMeditating` (@Published on an ObservableObject) and `applyTier` mutates CLLocationManager properties (`adjustPower`) from the background thread.
  - Fix sketch: Insert `.receive(on: DispatchQueue.main)` on both the batteryLevelDidChangeNotification (line 95) and thermalStateDidChangeNotification (line 99) publishers, matching the $isMeditating sink.
  - Verifier confidence: 0.88, 0.85

- AF13. **WalkSessionGuard JSON-encodes the entire walk (all route samples) and writes it to disk on the main thread every 30 s — every 10 s when battery is low or device is hot**
  - `Pilgrim/Models/Walk/WalkSessionGuard.swift:155` · lens: performance
